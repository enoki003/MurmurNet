## 2. ThreadPoolExecutorによる並列化

**問題点**

* 現在はAgent実行もSelf-Diagnosisも逐次実行しているため、エージェント数やステップ数が増えると遅くなります。

**改善案**

* `concurrent.futures.ThreadPoolExecutor` を使い、各 Agent の `generate_utterance()` や埋め込み計算を**同時並列**化します。
* GILの影響を抑えるため、I/O待ち（APIコールやディスク読み書き）がある部分を並列化すると特に効果的です。

```python
with ThreadPoolExecutor(max_workers=N_THREADS) as exe:
    futures = [exe.submit(agent.generate_utterance, snapshot) for agent in agents]
    for f in as_completed(futures):
        result = f.result()
        blackboard_buffer.append(result)
# 全部終わったらバッファをまとめて黒板に反映
```

# MurmurNetとBoids型エージェント制御のステップ同期アーキテクチャ設計

## 1. ステップ同期制御と軽量並列処理

MurmurNetにおいて複数エージェントの**発言生成**や**意見更新**を効率良く行うために、**ステップ同期型**のアーキテクチャを採用します。ステップ同期とは、全エージェントが短い離散時間ステップごとに同期して状態を更新する方式です。エージェント同士の相互作用が頻繁なマルチエージェントシステムでは、この同期型シミュレーションがよく用いられます。各ステップの開始時点で全エージェントの前ステップまでの状態が確定し、その情報に基づき一斉に行動計算を行うことで、イベントが密な状況でも安定した結果を得られます。

各ステップ内の処理では、**軽量並列処理**によってCPUコアを有効活用します。具体的には、各エージェントの発言生成タスクをスレッドプールで同時並行的に実行します。Pythonであれば`concurrent.futures.ThreadPoolExecutor`などを用いてエージェントごとの処理を並列化できます。これにより、複数エージェントの発言生成を同時に進め、ステップ全体の処理時間を短縮します。各タスク（エージェントの発言生成関数）は独立しており、他のエージェントとデータ競合しないように設計することで、スレッド間のロックを極力排除しつつCPUの複数コアを活用できます。以上のように、ステップ同期で一斉に処理しつつ、ステップ内ではエージェント処理を並列実行する構成にすることで、ビジネスPCレベルのCPU環境でもスループットを確保した**軽量実行**が可能となります。

## 2. 黒板型共有メモリとスナップショット戦略

エージェント間の情報共有には**Blackboard（黒板）**モデルを採用します。黒板とは全エージェントが読み書きできる共有データ構造であり、各エージェントは黒板上の情報を監視し、それに基づいて動作して結果を書き戻すことで間接的に協調します。本設計では、この黒板をベースに**スナップショット戦略**を導入します。各ステップの冒頭で黒板の内容を**スナップショット（読み取り専用コピー）**として作成し、エージェントはそのコピーを参照します。これにより、エージェントは現在の共有状態を**不変オブジェクト**として安全に読み取ることができ、ステップ内で他スレッドと衝突することなく発言生成処理を進められます。

各エージェントはスナップショットから自分に関連する情報（他エージェントの最新意見や議論コンテキストなど）を読み取り、**非同期**に発言内容や意見更新を決定します。エージェントの処理結果（発言テキストや更新後の意見値など）は、一時的な**発言リスト**や**更新キュー**に書き出され、直接黒板の本体を変更しないようにします。全エージェントの発言生成が完了したら、主スレッドでこれらの結果を集約し、**一括して黒板へ書き込み適用**します。この一括適用により、あるステップ内での状態更新は論理的に同時に起こるようになります。つまり、各ステップ間では黒板状態が不連続にジャンプし（ステップ境界でのみ変化し）、ステップ内では読み取り専用で不変となるため、並行処理による競合や読み書きの不整合が起こりません。

このスナップショット戦略により、排他制御（ロック）なしでもデータレースを防ぎ、**決定性**を確保できます。全エージェントが同一の入力（スナップショット）に対して処理を行い、その結果をまとめて適用するため、実行順序によらず同じ結果が得られます。また、各エージェント処理中に黒板の内容が変化しないため、古い情報と新しい情報が混在する不整合も避けられます。以上の仕組みにより、黒板モデルの柔軟性を保ちつつ、安全で効率的な並行実行を実現します。

## 3. Self-DiagnosisモジュールとBoidsAgentFactoryの同期処理

ステップ同期アーキテクチャでは、**Self-Diagnosis**モジュールと**BoidsAgentFactory**についても各ステップ単位で処理を行い、他の処理と明確に同期を取ります。これらは議論全体を管理・調整する重要なコンポーネントであり、エージェントの発言処理と並行して実行すると競合や非決定性を生む可能性があります。そのため、本設計では以下のようにステップ境界での実行とします。

* **Self-Diagnosis（自己診断）**: これは議論の進行状況を解析し、停滞や収束の度合いを判定するモジュールです。各ステップの終了時（黒板に全エージェントの発言・意見更新が反映された後）に、最新の黒板状態を**読み取って評価**します。Self-Diagnosisは例えば各エージェントの意見分布の変化量や議論内容の新規性などを分析し、「議論が活発か」「行き詰まりつつあるか」を判断します。この判定結果はフラグや評価値として**黒板上**またはコントローラに渡され、次ステップの制御に利用されます。Self-Diagnosisの実行自体もステップ内の他処理と排他的に行われ、**全エージェントの発言結果を確定**させてから評価するため、常に確定した情報に基づく安定した判定が可能です。

* **BoidsAgentFactory**: これはBoids型自己増殖エージェント制御に関するファクトリモジュールで、必要に応じて新規エージェントの生成や既存エージェントの消滅（退出）を管理します。BoidsAgentFactoryも各ステップの終わりに呼び出され、最新の黒板状態とエージェント集合を**読み取り**ます。そして、あらかじめ定義したルールに従い、エージェント数や構成を調整します。例えば、「ある意見を持つエージェントが一定以上増えたらそのクラスターを代表する新エージェントを複製する」「多数派と少数派の距離が遠い場合に媒介するエージェントを追加する」といった自己増殖・調整処理を行います。Boidsアルゴリズム由来の挙動として、エージェント群全体が**分離・整列・結合**といったルールで意見空間上で群れを成すことを目指すため、エージェントの増減もそれに沿って調節されます。BoidsAgentFactoryの処理結果（新規エージェントの生成や削除）は、次のステップ開始前に黒板とエージェントリストに**一括適用**されます。

上記のように、Self-DiagnosisとBoidsAgentFactoryはいずれも**ステップ境界**で順次実行されます。具体的な順序としては、「エージェントの発言更新 → 黒板更新 → Self-Diagnosis判定 → BoidsAgentFactory処理 → 次ステップへ」というシーケンスになります。こうすることで、例えばBoidsAgentFactoryがエージェントを追加・削除する際も他の処理との**競合を回避**できます。Self-Diagnosisの判定もそのステップ内で唯一の確定した情報に基づくため、再現性のある結果を得られます。全体として、これら管理系モジュールもステップ同期のフレームワークに組み込み、**状態変化の原子性**と**判定の確定性**を保証しています。

## 4. 議論ステップ数の決定戦略の比較

ステップ同期型の議論シミュレーションでは、**何ステップで議論を終了させるか**が設計上の重要なポイントです。本設計では以下の3つの方針を考慮し、それぞれ**CPU実行性能**・**議論品質**・**設計実装の複雑性**の観点から利点と欠点を整理しました。

### (a) **ステップ数固定**（例：5ステップで打ち切り）

**概要:** あらかじめステップ数を固定し、その回数だけ議論を進めたら強制的に終了する方式です。例えば「全5ステップ」と決めておけば、5サイクルの発言更新で議論を完結させます。

* **CPU性能:** 固定ステップ数の場合、実行するステップ数が予め決まっているため**計算量を見積もりやすい**という利点があります。5ステップで終えるならエージェント数や1ステップ当たり処理量に応じた時間で確実に終了し、最悪ケースが限定されます。したがって**リアルタイム性能の保証**がしやすいです。一方で欠点としては、議論が早期に収束して不要なステップが残っていても実行を続けるため**無駄な計算**が発生しうる点があります。逆に必要以上に短い固定値にしてしまった場合、議論が十分進まないうちに終了してしまうリスクもあります。この場合は**議論品質**の低下にもつながりますが、性能面でも本来もう少し進めれば得られたはずの情報を活用できない非効率が生まれます。

* **議論品質:** 固定ステップ方式では、議論の深さ・結論の妥当性が設定ステップ数に大きく左右されます。**利点**として、全エージェントが平等に発言できるステップ数を保証しやすく、議論のフォーマットを統一しやすい点が挙げられます。例えば5ステップなら「序論→展開→深掘り→収束→結論」といった構成にあらかじめ合わせることも可能です。しかし**欠点**として、実際の議論展開に即していない固定打ち切りは、議論が白熱して新たな論点が出ている途中でも強制終了する可能性があります。その結果、有益な意見交換を途中で止めてしまい**議論が不十分**になる懸念があります。また逆に、早々に結論が見えて参加者の意見も収束してしまっているのに、形式的にステップを消化するだけの冗長な発言が続くケースもありえます。このように固定ステップでは議論品質がケースによっては最適化されず、**柔軟性に欠ける**点がデメリットです。

* **実装複雑性:** 実装面では最も**シンプル**なアプローチです。ループを決まった回数まわすだけで良く、停止判定のロジックも不要です。したがって**実装コストやバグリスクは最小**となります。開発・テストもしやすく、議論ステップ数に関する追加のパラメータ調整も固定値を変えるだけで済みます。複雑性という観点では利点ばかりですが、その反面、上記のような品質・性能面での柔軟性不足をソフトウェア側で解決できないため、シナリオデザインや事前のステップ数設定に人手で調整が必要になる可能性があります。

### (b) **意見分布の多様性・変化量に応じた動的ステップ数**

**概要:** 各ステップ終了時にエージェントの**意見分布**やその**変化量**を分析し、議論がまだ進展しそうかどうかで次ステップを続行するか終了するかを判断する方式です。例えば「各エージェントの意見ベクトルのばらつき（分散）が一定以下かつ直近の変化が小さくなったら終了」といったルールを設定します。

* **CPU性能:** 動的ステップ数では終了するステップがケースバイケースで変わるため、最悪の場合のステップ数が読みにくくなります。**利点**として、議論が早期に収束した場合には予定より少ないステップで終了できるため計算資源を節約できます。無駄な追加ステップを実行しないので**平均的には高速化**が期待できます。一方**欠点**として、議論が難航した場合には固定ステップ以上に長く続行する可能性があり、最悪シナリオでは非常に多くのステップを消費してCPU負荷が増大する恐れがあります。また各ステップで分布計算や変化量計算といった**オーバーヘッド**が若干発生しますが、これはエージェント数や意見次元がそれほど大きくなければ無視できる程度です。総じて、性能面ではケースによって**ばらつき**が出る（ベストケースでは効率的、ワーストケースでは非効率）という特徴があります。

* **議論品質:** このアプローチは**議論内容に応じて柔軟に長さを調節**できるため、品質向上に寄与する可能性が高いです。利点として、エージェントの意見が十分に収束・合意形成されたと判断した時点で終了するので、**議論が煮詰まった後は冗長に続けない**適切な打ち切りができます。また、逆に意見が割れたまま・議論が深まっている途中で終わることを防ぎ、**必要なだけ議論を延長**できるため、より充実した結論や多面的な議論を引き出せます。しかし欠点として、自動基準の設定によっては**判断ミス**が生じるリスクがあります。例えばまだ議論の余地があるのに変化量が小さくなっただけで早期に終了してしまったり、逆に些細な意見変化を捉えて必要以上に議論を引き延ばしてしまったりする可能性があります。したがって議論品質は基準設定の妥当性に依存し、調整不良だとかえって質が下がる恐れもあります。

* **実装複雑性:** 固定ステップに比べると**実装は複雑**になります。まずエージェントの意見を定量化した**多様性指標（例：分散やエントロピー）**や**変化指標（例：前ステップとの差分距離の総和）**を計算する必要があります。また、それらとしきい値を比較するロジック、しきい値のチューニングパラメータが追加されます。シミュレーション開始前に適切な基準値を設定するための検証も必要です。さらに、この基準は議論の種類やエージェント数によって適切値が異なる可能性があり、場合によっては**動的なしきい値調整**も検討しなければなりません。こうした判断ロジックを実装・検証するコストが増えるのが欠点です。しかし構成自体はステップループに条件分岐を追加する程度で済むため、実装困難というほどではなく、**中程度の複雑性**と言えます。

### (c) **Self-Diagnosisによる停滞判定で自動終了**

**概要:** 前述のSelf-Diagnosisモジュールを活用し、**議論の停滞**や**収束**を検知したら自動的に議論を終了する方式です。Self-Diagnosisは単に意見の数値的な変化だけでなく、議論内容の重複や新規トピック有無など高度な解析も行えるため、人間の判断に近い形で「これ以上続けても新しい有益な意見は出ない」という状況を判定します。

* **CPU性能:** Self-Diagnosisを用いた場合も動的ステップ数の一種なので、終了タイミングはケースバイケースです。性能上の利点は、議論が停滞したら即座に終了できるため無駄な反復を避けられる点です。特に議論がループに陥ったり、同じ意見の焼き直しになっている場合、早めに打ち切ることで**計算資源の浪費を防げます**。また、Self-Diagnosis自体は各ステップで実行される追加処理になりますが、これはテキスト解析や統計計算など比較的軽量な処理で設計すれば（例えば発言履歴の簡単な特徴量計算程度）、全体のCPU負荷に大きな影響を与えません。一方欠点は、Self-Diagnosisのロジックが高度になるほど処理コストが上がる可能性です。例えば大規模言語モデルを用いて議論内容の意味解析を毎ステップ行うような場合、各ステップに**追加の推論コスト**が乗ってしまいます。しかし本設計ではCPU上で安定動作させる前提から、Self-Diagnosisも軽量なヒューリスティックや簡易モデルで実装する前提とすれば、性能面の問題は小さいでしょう。総じて、停滞判定による終了は平均的には効率化に繋がりますが、判定アルゴリズム次第で負荷が上下し得るという特徴があります。

* **議論品質:** Self-Diagnosisによる終了判断は、議論の文脈を考慮した**質的な判定**が可能な点が大きな強みです。利点として、単なる数値上の意見収束では捉えきれない「**議論の充実度**」を評価できます。例えば「新しい論点が3ステップ連続で出ていない」「参加者の発言が堂々巡りになっている」といったケースを検知して終了することで、議論が完全に停滞して消耗戦になる前に適切に打ち切れます。これは議論品質を保つ上で有効であり、**生産的でない議論を引き延ばさない**利点になります。また、必要な場合には固定ステップや単純なしきい値以上に議論を続行する判断も可能で、例えば「まだ意見の対立軸が残っている」とSelf-Diagnosisが判断すればステップ数が当初想定より増えても議論を続け、**より深い洞察や合意形成**を目指せます。欠点としては、Self-Diagnosisの判定精度が議論品質に直結するため、そのアルゴリズムの**信頼性**が問題となります。誤判定で早期終了してしまえば有望な意見の芽を摘むことになりますし、終了すべきタイミングを見逃すと冗長な議論が続いて品質低下を招きます。人間並みの判断基準を設計・実装するのは難易度が高く、性能とのトレードオフもあります。そのため品質面では**アルゴリズム設計次第**という側面が大きく、不確実性を伴う点が課題です。

* **実装複雑性:** このアプローチが最も**実装が複雑**になります。Self-Diagnosisモジュール自体の開発が必要であり、議論の停滞を検知するための**高度なロジックやモデル**を組み込む必要があります。例えばキーワードの多様性推移を見る、発言内容の類似度を計測する、あるいはMLモデルで「議論の健全度」をスコアリングする、といった実装が考えられます。これらは (b) の単純なしきい値方式に比べて**多くの試行錯誤**や**パラメータ調整**を要するでしょう。また、Self-Diagnosisと他コンポーネントとのインターフェース（例えば黒板上に評価結果を書き込む仕様など）も定義しなければなりません。さらに、実行順序や並行実行への配慮（ステップ同期内で確実に判定を終えるタイミングの保証など）も必要です。ただし、既にSelf-Diagnosisをシステムの一部として組み込んでいる前提ならば、その判定結果を利用するだけなので、**追加の終了判定ロジック自体はシンプル**とも言えます。いずれにせよ、高度な分析モジュールを必要とするため実装・チューニングの負担は最も大きくなります。

以上より、(a)固定、(b)動的しきい値、(c)高度判定による停滞検知の各方式には一長一短があります。**CPU負荷の予測しやすさ**では(a)が優れていますが、**議論の適応性**では(b)や(c)が有利です。(c)は最も人間らしい判断が可能な反面、**開発コスト**や**判定精度**の課題があります。システムの目的や制約に応じて、これらを組み合わせたり段階的に適用する（例えばまず最大ステップ数の上限を決めつつ、早期終了は(b)基準で判断し、さらに高度な終了判断は(c)で行う等）ことも検討に値します。

## 5. 統合アーキテクチャ設計とモジュール仕様

以上の検討を踏まえて、MurmurNetにBoids型自己増殖エージェント制御を組み込んだ**統合アーキテクチャ**を設計します。全体構成としては、**黒板**を中心に複数のエージェントと制御モジュールが接続された、黒板モデルベースの同期システムとなります。下図に主要なモジュール間の関係を示します。

&#x20;*MurmurNet+Boids統合アーキテクチャ図: 中央に共有メモリであるBlackboardがあり、Agents（複数のエージェント）がそれを介して間接通信する。各AgentはBoidsアルゴリズムに基づき意見形成し、Step Controllerの管理下で並列に動作する。Self-Diagnosisモジュールは各ステップでBlackboardを読み取り議論停滞を判定、BoidsAgentFactoryはBlackboardを読み取り必要に応じてエージェントを増減する。矢印は主なデータフローや制御フローを示す（実線はデータ読み書き、点線は制御信号）。*

上記のアーキテクチャ図では、Blackboard（黒板）が全情報のハブとして存在し、エージェントは直接互いに通信せずBlackboardへの読み書きを通じて協調しています。AgentsはBoids型の自己増殖エージェント群であり、Boidsアルゴリズム由来の分離・整列・結合の行動則を内部に持ちつつ、議論の発言生成タスクを担います。Step Controller（ステップ制御モジュール）は全体のタイムステップ進行を管理しており、各ステップの開始・終了、および各モジュールの実行タイミングを統括します。Self-Diagnosisモジュールは各ステップ終了時にBlackboard上のデータを解析し、必要ならStep Controllerへ「停滞検知」信号を送ります（例えば停止フラグを設定）。BoidsAgentFactoryは各ステップでBlackboard上のエージェント情報を読み、エージェントの追加・削除が必要かを判断してBlackboard（とエージェントリスト）を更新します。Step Controllerからは各エージェントへの並列タスク実行指示や、各評価モジュールへの処理指示が出され、全体の同期を取ります。

&#x20;*ステップ同期アーキテクチャの実行フロー: 各ステップの開始時にBlackboardのスナップショットを作成し、それを元に全Agentの発言生成処理を並列実行する。完了後、結果を集約してBlackboardを更新し、Self-Diagnosisによる停滞判定とBoidsAgentFactoryによるエージェント増減処理を行う。判定結果や更新内容を踏まえて次のステップに進むか終了するかをStep Controllerが決定する。このサイクルを停止条件に達するまで繰り返す。*

上記フロー図は、1ステップ内で行われる処理とステップ間の繋がりを示しています。まず**Step Controller**がステップ開始をトリガーすると、現在のBlackboard状態をコピーした**Snapshot**を全エージェントに配布します（読み取り専用）。各**Agent**はThreadPoolなどで並列に起動され、スナップショットデータに基づき発言内容や意見更新値を生成します。全エージェントの処理が完了すると、その**出力（発言群）**が集められ、**Blackboard**に一括適用されます（状態更新）。続いて、更新後のBlackboardを使って**Self-Diagnosis**が議論状態を評価し、停滞フラグなどの判定結果を出力します。さらに**BoidsAgentFactory**がBlackboard内のエージェント一覧やその意見分布を見て必要なエージェント増減を決定し、Blackboardに対してエージェント追加・削除を行います。最後にStep ControllerがSelf-Diagnosisの結果（例えば停滞フラグが立ったか、または既定ステップ数に達したか等）を確認し、次のステップを実行するかループを抜けるかを判断します。次のステップに進行する場合は再び最新のBlackboardからSnapshotを作成し、同様のサイクルを繰り返します。これにより、前節で述べた(a)(b)(c)いずれの停止条件にも対応できる柔軟なループ制御になっています。

最後に、本アーキテクチャを構成する**各モジュールの役割**と**主な仕様**をまとめます。

* **Blackboard（黒板）:** 議論状態のグローバル共有メモリ。全エージェントの現在の意見や発言履歴、議論テーマ、Self-Diagnosis結果（停滞フラグ等）などを格納します。データ構造はスレッドセーフに設計され、各ステップ内では読み取り専用で提供されます。具体的には、辞書やクラスオブジェクトで意見ベクトルやテキストログを保持し、Snapshotはそのディープコピーとして実装します。書き込みはステップ終了時に一括して行われます。

* **Agent（エージェント）:** 個々の議論参加者を表す自律エージェント。Boids型自己増殖制御の一部として、エージェントごとに**意見状態**（例えば数値ベクトルやカテゴリ）のほか、発言生成のための内部モデル（簡易なNLPモデルやルールベースでも可）を持ちます。各Agentは`generate_utterance(snapshot)`のようなステップ処理関数を持ち、引数として与えられるSnapshot（ブラックボードのコピー）から他者の意見や直近の発言を参照し、自分の発言内容と自分の意見状態更新を決定します。生成された発言はテキスト文字列等で表現され、意見状態更新は内部の状態を書き換えるかブラックボード更新用の差分として出力します。Boidsアルゴリズムのエッセンスとして、各エージェントは周囲（他エージェント）の意見に影響を受けてわずかに自分の意見をシフトさせたり、逆に一定距離を保ったりといった調整を行います（分離・整列・結合のルールを意見空間上で適用するイメージ）。自己増殖（複製）に関しては、エージェント自身は行わずBoidsAgentFactoryが集中管理しますので、Agent側は増減のトリガーとなる情報（例：自分のクラスタの勢力が拡大した等）をブラックボードに書き出す程度です。各Agentは軽量に設計し、CPU上で並行実行しても処理落ちしない程度の負荷（例えばシンプルなルールベース対話や小規模言語モデル推論）に抑えます。

* **ThreadPoolExecutor（並列実行基盤）:** エージェントの`generate_utterance`処理を同時並行で実行するためのスレッドプールまたはタスク実行基盤です。Step Controllerから各Agentの処理がサブミットされ、内部のワーカースレッドで非同期に実行されます。プールサイズはCPUコア数やエージェント数に応じて設定し、エージェント数が多い場合はキューイングされるか、必要に応じて動的拡大できるようにします。重要な点は、各Agent処理は状態をSnapshotから読むだけで書き込みを伴わないため**スレッド間の干渉が無い**ことです。この前提により、ロックフリーで各タスクを安全に並行実行できます。実装的にはPythonの場合GILの制約がありますが、I/O待ちや軽い計算であればThreadPoolで十分対応可能です（重い計算ならマルチプロセスも検討）。また、並列処理中に例外が発生した場合でも他のスレッドに波及しないようにし、全スレッドの完了を待ち合わせてから次の処理に進みます。

* **Step Controller:** ステップ同期全体を取り仕切るコントローラです。おおまかな擬似コードは以下のとおりです：

  ```
  initialize_blackboard()
  step = 1
  while step <= max_steps:
      snapshot = deepcopy(blackboard)
      futures = [ThreadPool.submit(agent.generate_utterance, snapshot) for agent in agents]
      wait_until_all_complete(futures)
      results = collect_results(futures)
      blackboard.apply_results(results)  # 発言・意見更新を一括反映
      diagnosis = SelfDiagnosis.check(blackboard)  # 停滞判定
      if diagnosis.stagnation_flag:
          break  # 停滞と判断 -> ループ終了
      BoidsAgentFactory.update_agents(blackboard, agents)  # 必要ならエージェント増減
      step += 1
  end while
  ```

  Step Controllerは上記のように各コンポーネントを順序通り呼び出し、停止条件に達したらループを抜けます。停止条件には最大ステップ数やSelf-Diagnosisの結果、その他任意の基準を組み込めます。Step Controller自体は単一スレッドで動作し、各ステップ内では主にタスクのスケジューリングと結果収集に徹します。これにより、全体の**同期ポイント**がこのコントローラで一本化され、複雑な並行同期のバグを防ぎます。

* **Self-Diagnosis:** 議論モニタリングと終了判定を担うモジュールです。仕様として、Blackboard上のデータ（全エージェントの意見や発言履歴など）を入力とし、**議論状態評価指標**を計算します。例えば「意見分布多様性スコア」「直近Nステップの新規キーワード出現数」「賛否の拮抗状況」などの評価値を算出し、Blackboardに書き込みます。また特定条件下では`stagnation_flag = True`のような終了シグナルもBlackboardまたはStep Controllerに通知します。Self-Diagnosisは各ステップ**後**に実行され、ステップ内のエージェント発言内容も解析できます。軽量化のため、テキスト分析にはキーワードマッチや簡単なベクトル化＋類似度計算程度に留め、大規模なモデル推論は避けます。将来的に高度化する場合でも、モジュールを差し替えられるようインターフェース（入力はBlackboard全体、出力は評価結果オブジェクトなど）を明確に定義しておきます。

* **BoidsAgentFactory:** エージェント集合の動的制御を行うモジュールです。各ステップ終了時に呼ばれ、Blackboardから**現在のエージェント情報**（人数、各エージェントの意見値やクラスタ構造など）を取得します。Boidsアルゴリズムの**結合（集団化）ルール**に倣い、例えば意見が類似したエージェント同士をグルーピングして中心に新エージェントを追加する、といった操作を行います。また**分離ルール**として、意見が極端に偏ったエージェントがいたらそのエージェントを除去する、**整列ルール**として、中庸な意見のエージェントを新規投入して全体のバランスを取る、等の戦略が考えられます。具体的な仕様はシナリオによりますが、一例として「最大エージェント数上限まで、最大意見クラスタに属する意見を複製する新エージェントを追加する」「一定ステップ発言していないエージェント（休止状態とみなす）を削除する」などが挙げられます。BoidsAgentFactoryはこれらルールに基づきエージェントリストを更新し、変更があればBlackboard上のエージェント情報テーブルも修正します。なお、エージェントの生成には固有のID付与や初期意見値の設定を行い、削除時にはBlackboard上の関連情報も消去します。Step Controllerとは、処理完了後に「エージェント増減あり/なし」のフラグやログを渡しておき、デバッグや分析で追跡できるようにします。

以上が本設計の詳細です。CPU上で安定動作させるための軽量並列と同期制御を組み合わせ、ブラックボードによる緩やかな協調動作を実現しています。Boids型のエージェント制御を組み込むことで、各エージェントがシンプルなルールから群集的な振る舞いを見せ、議論に**創発的な多様性**をもたらします。ステップ数制御についても複数の基準を組み合わせ可能な柔軟性を持たせ、議論の質と効率のトレードオフに対応しました。本設計ドキュメントが、軽量なCPU推論環境で複数エージェントAIを用いた議論システムを実装する際の指針となれば幸いです。各コンポーネントの実装を進める際には、本書のアーキテクチャ図やフロー図を参照し、モジュール間インターフェースとステップ同期のタイミングを遵守することで、整合性の取れたシステムを構築できるでしょう。
